# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A parody web app: an "AI longevity scan" that runs a mock-clinical lifestyle quiz, fakes an AI analysis, reveals a predicted date of death, and funnels the user into buying a (real) fitness guide. The death estimate is fictional; it exists as marketing. This framing matters: the experience is deliberately deadpan-credible, and the "this is a parody" disclosure lives only in the legal pages, never in the main UI.

## Commands

```bash
npm run dev        # local dev server at http://localhost:3000
npm run build      # production build (also runs tsc + eslint)
npm run lint       # eslint
npm test           # vitest run (engine unit tests in lib/*.test.ts)
npm run test:watch # vitest watch
npm run e2e        # Playwright smoke test (drives the full scan flow; reuses a running dev server)
```

Run a single engine test:
```bash
npx vitest run lib/longevity.test.ts -t "best-case answers push life expectancy up"
```

After UI or copy changes, the verification loop that has caught real issues is: `npm test && npx tsc --noEmit && npm run lint && npm run build`, then `npm run e2e`.

## Stack and non-obvious constraints

- **Next.js 16 (App Router), React 19, TypeScript, Tailwind v4, Motion (`motion/react`), Geist fonts.** Next 16 has breaking changes vs. older training data — see `AGENTS.md` / `node_modules/next/dist/docs/` before writing Next-specific code.
- **shadcn/ui here is built on `@base-ui/react`, not Radix.** Consequences that bite:
  - `Button` has no `asChild`. To render a link as a button, use `buttonVariants()` + `cn()` on a `<Link>` (see `app/page.tsx`).
  - `RadioGroup` uses `onValueChange` (not `onChange`) and marks selection with a `data-checked` attribute (selected-state styling keys off `has-[data-checked]`, see `components/quiz-step.tsx`).
- **No em-dashes (`—` or `–`) in any user-facing string.** This is enforced; use hyphens, commas, or periods. UI work is expected to go through the `design-taste-frontend` skill (per the user's global config).
- **The funnel copy is US-native and imperial.** All user-facing copy (quiz, analyzing log, report, pitch, reviews, the delivered guide) targets a mass-market US reader: punchy, plain, deadpan-clinical, never insulting. Use imperial units (lbs, inches, Fahrenheit) and US spelling/terms (takeout not takeaway, ground beef not mince, zucchini not courgette, fiber not fibre). Keep new copy matching this voice. Note: recipe ingredient weights are still in grams (and the ingredient-parsing regex in `build-guide.ts` keeps `g|ml|litre|liter` tokens, do not remove them).
- **Theme is a single locked dark "diagnostic monitor" mode.** Color tokens are defined in `app/globals.css` via Tailwind v4 `@theme` as `--color-monitor-*` (utilities like `bg-monitor-panel`, `text-monitor-accent`). One brand accent (teal `monitor-accent`); red (`monitor-alert`) is reserved for mortality/risk data only. The ONE sanctioned exception: two per-gender "trace channel" tints (`--color-monitor-trace-m` steel blue, `--color-monitor-trace-f` warm amber) used ONLY on the gender-targeted result-page elements (the report card's "Subject profile" chip + stakes-line border, and the pitch's "Calibrated: male/female profile" module). Do not use them anywhere else or as a second brand accent. shadcn semantic tokens (`--color-primary`, etc.) are mapped to the monitor palette in the same file, so do not delete them.

## Architecture (the parts that span multiple files)

**`lib/longevity.ts` is the single source of truth and the only place real logic lives.** It is a pure, deterministic, offline "model" (no network, no AI call). `computeResult(answers, today?)` returns the full `ScanResult`; `today` is injectable so tests are deterministic. Key behaviors that everything downstream depends on:
- `QUESTIONS` drives the quiz. Questions can branch (`showIf(answers)`) and can be collected-but-not-scored (`scored: false`, e.g. the goal question). `getActiveQuestions(answers)` returns the visible, ordered list. The quiz is **gender-aware**: the scored `sex` question gates a female hormonal-stage question (`hormonal_female`, scored, non-recoverable) and a male testosterone-signals multi-select (`hormonal_male`, scored, recoverable), which also drive gender-specific `outcomes`, `analysisSignals`, and gender-keyed result-page copy (the stakes line in `report-card.tsx`, the relevance line in `guide-pitch.tsx`). Keep that copy deadpan-clinical, never insulting.
- The estimate uses decimal, epidemiology-flavored weights, and negative factors run through a diminishing-returns curve (`NEG_CAP`, `NEG_SCALE`) so a worst-case profile bottoms out believably instead of free-falling. **Keep the estimate deterministic for a given set of answers** — only cosmetic things (the analysis log) may use randomness.
- It also derives the report and funnel inputs: `topRisks`/`strengths` (for the report), `recoverableYears` (measured as the dampened gain from neutralizing modifiable risks), results-based `outcomes` (goal-aligned), a per-user `modelConfidence`, and `analysisSignals(answers)` (answer-specific phrases for the analyzing screen). It is covered by `lib/longevity.test.ts`.

**`app/scan/page.tsx` is the client state machine** for the whole experience: `quiz -> analyzing -> result`. The quiz is dynamic — it tracks the current question by id and recomputes the active list from `getActiveQuestions(answers)` each render, so branching follow-ups appear/disappear live. The result phase renders the shared **`components/result-view.tsx`** (`SaleProvider` wrapping `ReportCard` + `GuidePitch` + `ResultStickyBar`, computing the result from `answers`); the result-page CTAs start guide generation directly (no smooth-scroll jump) via the shared `components/use-checkout.ts` hook, threading `answers` down to `CheckoutButton`/`ReportCard`/`ResultStickyBar`. `ResultView` is deliberately shared so the live scan AND the emailed result link (`app/result/[token]`, see the email-wall section) render an identical reveal + pitch.

**Pricing/urgency is a React context, `components/sale-context.tsx`.** `SaleProvider` mounts on the result page (the result phase of `app/scan/page.tsx`), so the countdown starts when the result lands. The deadline is **persisted per visitor in `localStorage`** (key `sw_offer_deadline`), so a refresh or a fresh scan does NOT reset it: the window is a real, one-time countdown and the price genuinely stays up once it ends. This is deliberate, the urgency must stay truthful (no fabricated scarcity, no secretly-resetting timers). When the timer hits zero the price rises (`PRODUCT.price` -> `PRODUCT.expiredPrice`). `useSale()` exposes `remaining` (null when there is no provider, e.g. the evergreen `/guide` page), `expired`, `price`, `expiredPrice`, `listPrice`. Every price/timer display (`ReportCard`'s first-ask, `GuidePitch`'s `OfferModule`, `ResultStickyBar`, `CheckoutButton`) reads `useSale()` — never hardcode the price.

**The landing + result pages are a conversion funnel (truthful price psychology).** `app/page.tsx` (landing) is now a long-form sales page (problem -> agitation -> solution), built mostly above the fold: an extreme, specific, emotional mortality hero (huge headline + pain subhead + dominant `/scan` CTA + proof microbar + honest trust line "Takes 90 seconds. No card to see your result." — never claim "no email", the result is email-gated), a pain/agitation block naming the slow decline, a how-it-works stepper, a locked/redacted death-date panel (curiosity gap), **social proof on the landing** (`components/landing/proof.tsx`: aggregate stat bar + detailed reviews incl. the 4-star critical one, reading the same `lib/guide/testimonials.ts` data; numbers stay consistent with `components/reviews.tsx` — 4.8 avg / 2,847 readers), a loss-framed stakes block, a tight FAQ, and repeated `/scan` CTAs. Section components live under `components/landing/` (`proof.tsx`; `reveal.tsx` = restrained scroll-in motion that respects reduced-motion). If you show a price on the landing, resolve currency server-side and use `lib/money.ts` `formatMoney` like `app/scan/page.tsx`; never hardcode "$". The result page is ordered for hierarchy: `ReportCard` (`components/report-card.tsx`) keeps the death-date reveal dominant then makes a lean first-ask (loss line + price + live countdown + CTA) with condensed risk drivers; `GuidePitch` (`components/guide-pitch.tsx`) runs promise -> tight value stack (`INCLUDED`) -> concentrated social proof -> one dominant `OfferModule` (deal anchor `stackValue`/`listPrice`/`price`, live countdown, price-jump warning, loss framing) -> close. The levers are loss aversion (the death date), deal anchoring, the real persistent countdown, and sunk-cost, all kept honest. The parody disclosure stays in the legal pages only.

**The product is a comprehensive per-user 90-day program + asset kit, generated deterministically server-side (no AI call; the post-purchase AI Deepscan described below is the one exception).** Clicking buy calls the `beginCheckout(answers, { expired, currency })` server action (`lib/guide/checkout.ts`); when `STRIPE_SECRET_KEY` is set it creates an `awaiting_payment` order and redirects to hosted Stripe Checkout, and when it is unset it falls back to `startGuideGeneration(answers)` (`lib/guide/start.ts`, direct generate, what dev/e2e use). Either way the order is stored in `lib/guide/orders.ts` (Supabase when `SUPABASE_URL`/`SUPABASE_SERVICE_ROLE_KEY` are set, else an in-process store for dev/tests) and the buyer lands on `/guide/[token]`. The guide is built by `lib/guide/build-guide.ts`, a deterministic offline engine that branches on the answers (training level, past injury, diet, goal, sleep/stress, barrier) into the rich `GuideDoc` shape (`lib/guide/schema.ts`). The program is framed as **90 days**: the week-by-week schedule (the `weeks` array, still 8 entries) is the detailed first 8 weeks, and `programArc` is the full 90-day roadmap (Foundation 1-4, Build 5-8, Push 9-12, Maintenance beyond) with monthly progress reviews. Alongside it the engine derives a personalized **`yourNumbers`** dashboard (estimated starting bands + goal targets + a 90-day milestone map, framed as estimates/goals, never as measured biometrics), four **`bonusModules`** (Plateau Protocol, Travel kit, Supplement Truth, Life After Your 90 Days — each a `DeepDive`), **`trackers`** data (aisle grocery list + daily checklist), a **`recipeBank`** (recipes selected by goal/diet with estimated macros + an aisle shopping list, from `lib/guide/recipes.ts`), an **`exerciseLibrary`** (full entries for every prescribed movement, from `lib/guide/exercises.ts`), and **`scienceNotes`** (per-lever mechanisms with real references, from `lib/guide/science.ts` — do not fabricate citations; only Mandsager et al. JAMA Network Open 2018 is named, everything else is general phrasing). It runs on the first status poll (`app/api/guide/[token]/status`), so it never waits on background timing; `/guide/[token]` shows a polling "building" screen then the guide (`components/guide/guide-view.tsx`), whose top "kit" block offers five downloads. **Payment is real (Stripe hosted Checkout).** A verified payment flips the `awaiting_payment` order to `generating` via `markPaid` (idempotent: webhook `app/api/stripe/webhook/route.ts` is canonical, success-page `?session_id=` verification in `app/guide/[token]/page.tsx` is the fallback; both call `fulfillPaidOrder` in `lib/guide/fulfill.ts`, which emails the buyer their link via Resend `lib/email/send.ts`). `generateGuide` then runs as before. Server clamps the charged amount from `chargeAmountMinor(currency, expired)` (never trusts the client); an unpaid order never renders the guide. **Pricing is multi-currency.** `lib/product.ts` holds `PRICES` keyed by currency (USD/GBP/EUR/CAD/AUD: `price`/`expiredPrice`/`winbackPrice`/`listPrice` + a `rate`); the value stack `INCLUDED[]` is authored in USD and localized via `localizedValue`/`stackValueFor` (USD stack = 571, incl. the $79 AI Deepscan line, locked by `lib/product.test.ts`). Win-back tiers are floored so every purchase clears at least $9 USD profit after Stripe fees (~2.9% + $0.30) and the Deepscan's ~$0.03 of Haiku tokens. Currency is geo-resolved server-side (`x-vercel-ip-country`, `?cur=` override) in `app/scan/page.tsx` and threaded through `SaleProvider`; every price display reads `useSale()` (`symbol`/`price`/`stackValue`) and `lib/money.ts` `formatMoney` — never hardcode `$`. Every `INCLUDED` line must map to something actually delivered: do not re-add a "community" or coaching promise (there is none), and keep the 90-day framing (no stray "8-week" umbrella copy).

**An email wall gates the result, building the marketing list.** Between the analyzing screen and the death-date reveal, `app/scan/scan-flow.tsx` shows `components/email-gate.tsx` (the `capture` phase): email is REQUIRED to unlock the result, with a separate UNTICKED "tips and offers" consent checkbox (UK GDPR/PECR). Submit calls the `captureLead` server action (`lib/marketing/capture.ts`), which upserts the lead via `lib/marketing/subscribers.ts` (Supabase `subscribers` table when configured, else in-memory, mirroring `orders.ts`) and sends the free personalized report email (`emails/report-email.tsx`, a `@react-email/components` template rendered to HTML and sent through `sendReportEmail` in `lib/email/send.ts`, no-op without `RESEND_API_KEY`). The report email is the service the user asked for and sends regardless of consent; only `consented` subscribers get the marketing drip. **The report email's primary button links to the buyer's OWN result, not a blank scan**: `captureLead` sets `offerUrl` to `/result/<signed-token>` where the token is a signed HMAC over email + currency (`lib/marketing/email-links.ts` `signResultToken`, 7-day TTL, mirrors `winback.ts`). `app/result/[token]/page.tsx` (server, `force-dynamic`) verifies the token, loads the subscriber's STORED answers via `getSubscriberByEmail`, and re-renders the shared `components/result-view.tsx` (no re-quizzing); a bad/expired/unknown token redirects to `/scan`. Every marketing email also carries a working one-click unsubscribe (see the drip section). This deliberately overrides the old "no email to run the scan" promise; `/privacy` and `/terms` were updated to match. The e2e smoke test passes the gate via `passEmailGate`. The report email sends on EVERY scan (it is the transactional result the user asked for) EXCEPT when the recipient's daily cap is spent. `captureLead` is a public, unauthenticated endpoint, so it bounds input (`AnswersSchema` caps + email max length) and rate-limits via `lib/marketing/rate-limit.ts`: a DURABLE per-IP throttle (20/10min) and a DURABLE per-EMAIL send cap (3/24h), both backed by the Supabase `rate_limit_hit` atomic function (`supabase/migrations/20260605160000_rate_limits.sql`) with an in-process fallback when Supabase is unset or the RPC errors. The per-email cap is the real anti-bomb control: keyed on the recipient, it bounds how many reports any one address can receive regardless of attacker IP rotation, and over-cap captures still return `ok` but skip the send (no send oracle). `rateLimitDurable` is the function to use for anything that must hold across serverless instances; the legacy sync `rateLimit` is per-instance only. Optional future hardening (NOT yet done, no CAPTCHA by choice): a CAPTCHA/Turnstile check would let the per-email cap be looser, double opt-in proves ownership outright. Tracked in `AppState.md`.

**The marketing drip + win-back are a scheduled email sequence.** Consented subscribers are enrolled by `captureLead` into a 3-step sequence: the instant report email (inline at capture), then a value/objection email at +1 day and a win-back offer at +2 days, queued in `lib/marketing/email-jobs.ts` (`email_jobs` table, else in-memory; idempotent per email+kind). A Vercel Cron (`vercel.json` -> `app/api/cron/email-drip/route.ts`, secured by `CRON_SECRET`) runs `processDueJobs` (`lib/marketing/drip.ts`), which suppresses anyone unsubscribed, no longer consented, or who has since purchased (`hasPaidOrderForEmail` in `orders.ts`), then sends `emails/value-email.tsx` / `winback-email.tsx`. The value email's CTA also points at the subscriber's `/result/<signed-token>` (same `signResultToken`). The win-back email links to `/api/winback/[token]`, where a signed token (`lib/marketing/winback.ts`, HMAC over email+currency+expiry, 7-day TTL) unlocks a server-clamped checkout at `winbackPrice` (`winbackAmountMinor`) using the subscriber's STORED answers, so the client supplies nothing and cannot self-mint a discount. Verified live at the old tier: win-back session = 900 (USD $9); the tier is now $10 (1000 minor units, raised 2026-06-12 for the $9-profit floor) and has not been re-verified live. **Unsubscribe is real and one-click**: every marketing email footer renders a live `Unsubscribe` link to `/api/unsubscribe/<signed-token>` (`signUnsubscribeToken` in `lib/marketing/email-links.ts`, HMAC over email, ~1-year TTL), and the marketing sends carry RFC 8058 `List-Unsubscribe` + `List-Unsubscribe-Post: One-Click` headers (`lib/email/send.ts`). The route (`app/api/unsubscribe/[token]/route.ts`, GET + one-click POST) verifies the token, calls `markUnsubscribed`, and returns an on-brand confirmation page (invalid token -> "link no longer valid"); it never exposes the address, and `processDueJobs` already suppresses unsubscribed/non-consented/purchased addresses. The report email (transactional) carries the same unsubscribe link for hygiene since the address may be on the drip. KNOWN GAP (not yet fixed, see `AppState.md`): `beginCheckout` is itself a client-invoked `"use server"` action whose options accept `winback?: boolean`, and that flag is NOT enforced server-side, so a crafted request can pass `winback: true` to get `winbackPrice` without a token. The fix is to move session creation into a non-action helper and drop `winback` from the client-reachable action.

**The AI Deepscan is the ONLY real AI call in the app, and it is post-purchase.** Every purchase includes one Deepscan run: on `/guide/[token]` the buyer takes a full second assessment, a **28-question, sex-branched intake walked one question at a time** (wizard UX like the main quiz: intro -> progress-bar steps grouped into seven sections [body composition, heart and vitals, fueling, recovery and sleep, stress and mind, movement, hormones and medical] -> analyzing -> report; `lib/deepscan/questions.ts`, branching keyed on the scan's `sex` answer), and `runDeepscan` (`lib/deepscan/run.ts`) has Claude Haiku (`claude-haiku-4-5-20251001`, structured JSON output via `output_config`, ~2K input + 6K output cap = ~$0.03 of tokens per run, budgeted in `lib/product.ts`) write a marker-by-marker readout validated against `lib/deepscan/schema.ts` (summary, 8-12 markers with optimal/watch/flag status, 6-7 narrative sections **each with its own concrete `actions` list**, ordered priorities, disclaimer). Live-verified 2026-06-12 (real Haiku run, ~50s; UI says "under a minute"). The e2e suite forces `ANTHROPIC_API_KEY=""` in `playwright.config.ts` so smoke runs never bill or slow on real AI calls. **Without `ANTHROPIC_API_KEY`, or if the AI call fails for any reason, the deterministic offline builder (`lib/deepscan/fallback.ts`) serves the report instead** - dev/e2e/CI run entirely on the fallback, and a paying buyer never sees a failure. The client entry is the `runDeepscanAction` server action (`lib/deepscan/actions.ts`): it enforces ONE run per order (a replay returns the stored record, so a purchase can never fund a second AI run), bounds input, and validates answers against the buyer's active question set. The record persists inside the order's `guide` jsonb (`GuideDoc.deepscan`, optional field; `updateGuide` in `orders.ts`) so no DB migration was needed. UI is `components/guide/deepscan-section.tsx` (intake form -> analyzing state -> report), rendered at the top of `guide-view.tsx`. Model output is em-dash-sanitized before storage (house style). Covered by `lib/deepscan/deepscan.test.ts` and the e2e purchase test. The Deepscan is also the first `INCLUDED` line ($79) and is named in the report-card first-ask, the pitch FAQ, the landing FAQ, and `PRODUCT.tagline` - if you remove or rename it, update all of those together.

**The downloadable kit is five `@react-pdf/renderer` documents, served by one route.** `app/guide/[token]/download/[asset]/route.ts` maps `asset` in `{ workbook | trackers | quickstart | recipes | exercises }` to a Document and filename (`renderToBuffer`, nodejs runtime); an unknown token/asset 404s, a not-ready order redirects to `/guide/[token]` (the building screen). The legacy `app/guide/[token]/pdf` route redirects to `/download/workbook`. The documents: `components/guide/guide-pdf.tsx` (the full **workbook**: cover, table of contents, every guide section incl. the 90-day arc, science, recipe bank, exercise library, and printable tracker grids), `tracker-pack-pdf.tsx` (printable trackers, incl. the 12-week measurement grid), `quickstart-pdf.tsx` (a one-page start-today summary), `recipe-book-pdf.tsx` (a focused recipe book), and `exercise-library-pdf.tsx` (a focused exercise manual). Shared font registration, the monitor palette `C`, the tracker render components, the content cards (`RecipeCard`, `ExerciseEntryCard`, `ScienceEntryRow`, phase/review rows), and `coverStyles` live in `components/guide/pdf-shared.tsx` and are imported across documents (do not duplicate them). **No `fixed` running header/footer in ANY of these documents**: react-pdf 4.x miscomputes layout for any `fixed` element across the many auto-broken pages and throws `unsupported number`, which would 500 the download route. Consequently the table of contents lists sections with no page numbers, and grids are plain flex `View`s (use `wrap={false}` only on small atomic blocks, never a whole grid). `lib/guide/pdf-render.test.ts` is the regression guard: it `renderToBuffer`s all five documents for sample guides and asserts a non-empty buffer. Verify any PDF change by running it.

**Social proof is fixed and shared.** Reviews (`components/reviews.tsx`, with a per-review star rating, verified-buyer badge, and an aggregate score) and at-home before/after transformation images (`components/transformations-gallery.tsx`) render on the result page (inside `GuidePitch`) and the building screen. The landing page ALSO carries social proof now (`components/landing/proof.tsx`, a landing-specific section reading the same `TESTIMONIALS`; the old "keep social proof off the landing" rule is retired). They all read from `lib/guide/testimonials.ts`: `TESTIMONIALS` carry `quote`/`name`/`meta`/`detail`/`rating`/`verifiedAgo`, and `TRANSFORMATIONS` reference the images (the `weeks`/`stat` timeframe fields were removed). The files in `public/transformations/` (generated with the Higgsfield CLI) are **static-imported** in `testimonials.ts`, not string paths, so Next fingerprints them by content hash and replacing a file busts the optimizer + browser cache automatically. Do not revert to `/transformations/x.jpg` strings, or swapped photos render stale. **Do not add result-speed timeframes** (e.g. "down 8 kg in 8 weeks", "by week three") to review copy, transformation cards, or the guide's results FAQ — they were deliberately stripped as unrealistic. Each of reviews and transformations carries a small asterisk "illustrative / AI-generated / results vary, not typical" disclaimer linking to `/terms`; the full disclosure lives in `/terms`.

**Legal pages carry the disclosure and the company identity.** `/privacy`, `/terms`, `/cookies` (under `app/`, rendered via `components/legal-page.tsx`) are linked from `components/site-footer.tsx`. The consumer **brand is Vivrun** (domain `vivrun.com`); the operating company is **ColrosaStudios LTD** (registered in England and Wales; governing law England and Wales), named across all three legal pages and the footer. "Vivrun" is the site/brand wordmark (uppercase `VIVRUN` in the footer + the PDF `BRAND` const, currently duplicated across the five PDF docs); the PRODUCT name `PRODUCT.name` ("The Second Wind Protocol") is separate from the brand. **Favicon + social images are file-based metadata in the monitor palette, using a "flatline pulse" mark** (a teal EKG line dropping to a red flatline): `app/icon.svg` (favicon), `app/apple-icon.tsx`, and `app/opengraph-image.tsx` + `app/twitter-image.tsx` (`next/og` `ImageResponse`; the pulse line is an embedded data-URI SVG `<img>` because Satori does not reliably render inline `<svg>` shapes). `app/layout.tsx` sets `metadataBase`/`openGraph`/`twitter` from `NEXT_PUBLIC_SITE_URL`. Note: while the site password gate (`proxy.ts` -> `lib/site-gate.ts`) is on, scrapers cannot fetch `/opengraph-image`; it becomes reachable once the gate is removed for launch. `app/terms/page.tsx` carries full coverage: the parody / "not a real AI or medical estimate / it is marketing for the real guide" disclosure (this belongs ONLY in the legal pages, never the product UI), plus health/fitness + assumption-of-risk, 18+, no-warranty, limitation of liability (capped at amount paid), the UK statutory-rights carve-out (death/personal-injury by negligence, fraud, consumer rights unaffected — this is what keeps the liability limits lawful), indemnity, and a Consumer Contracts Regulations 2013 digital-cancellation waiver. `app/privacy/page.tsx` is UK GDPR / DPA 2018 aware (data-subject rights + the ICO) and now covers the email-wall: it collects the visitor's email + scan answers to deliver the report (lawful basis: steps taken at your request / legitimate interest) and opt-in marketing on consent (withdrawable), with retention + unsubscribe described. **No contact emails exist yet**: the pages use clearly-bracketed `[company number to be added]` / `[registered office address to be added]` placeholders and say a support contact will be published — do not invent a company number, address, or email.
